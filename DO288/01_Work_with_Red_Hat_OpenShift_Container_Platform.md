### Red Hat OpenShift Container Platform (DO288) Master Exam Guide

| Objective | Instructor Expectations & Accurate Behavior | Essential `oc` Commands & Tasks | Exam / Practical Usage | Key Notes & Advanced Exam Mechanics |
| :--- | :--- | :--- | :--- | :--- |
| **Work with Red Hat OpenShift Container Platform**[cite: 1] | Understand architecture and how resources connect. Instructors expect you to know how to navigate the API natively without external documentation[cite: 1]. | `oc login -u <user> -p <pass> <url>`[cite: 1]<br>`oc api-resources`[cite: 1]<br>`oc explain deployment.spec.template.spec`<br>`oc get events` | Logging into the cluster, validating your context, and looking up resource definitions[cite: 1]. | **The Core Exam Lifeline:** You will not have access to external search engines during the exam. `oc explain` acts as your built-in documentation for YAML structure. If you forget how to define a volume mount or a probe in YAML, `oc explain` provides the exact syntax. Additionally, running `oc get events` is the optimal first step for troubleshooting, as it displays the cluster's real-time operations and errors. |
| **Create and work with multiple OpenShift projects**[cite: 1] | Treat projects as strict boundaries[cite: 1]. Projects are OpenShift's wrapper around standard Kubernetes namespaces, automatically injecting default ServiceAccounts and security policies. Deploying an app in the wrong project during an exam is a guaranteed failure for that task[cite: 1]. | `oc new-project <name>`[cite: 1]<br>`oc project <name>`[cite: 1]<br>`oc status`[cite: 1]<br>`oc delete project <name>` | Creating isolated namespaces and verifying your current active project before running commands[cite: 1]. | **Exam Trap:** Exams often require you to work across multiple different projects. A common mistake is scaling a deployment or linking a secret in the wrong namespace because you forgot to switch contexts. Always run `oc project <target-name>` as the absolute first step for any new exam task. `oc status` provides a high-level tree view of what is currently running in your active project[cite: 1]. |
| **Create and deploy single-container applications**[cite: 1] | Know how to deploy from existing images, source code, or YAML manifests, and understand the strict separation between internal cluster routing and external ingress. | `oc new-app <image> --name=<app-name> -e KEY=value`<br>`oc scale deploy/<name> --replicas=3`<br>`oc expose svc/<name> --port=<port>`<br>`oc create route edge --service=<name>`[cite: 1] | Rapidly deploying stateless web servers or database instances[cite: 1], injecting environment variables, and safely exposing ports to external users. | **Traffic Architecture:** External traffic hits the **Route** (hostname) ➔ OpenShift Router passes it to the **Service** (Internal Load Balancer with a stable IP) ➔ Traffic is distributed to the **Pods** (Ephemeral compute instances). You can set environment variables during creation with `-e`, or modify a running deployment using `oc set env`. Modifying an environment variable automatically triggers a new rollout of the application pods. |
| **Create and deploy multi-container applications**[cite: 1] | Decoupled components (such as a frontend UI and a backend database) must be deployed as *separate* Pods to scale independently. Multi-container Pods are strictly reserved for tightly coupled processes that must share local resources. | `oc new-app` (run for each distinct component)<br>`oc apply -f <manifest.yaml>`<br>`oc get pods -o wide`[cite: 1] | Deploying complex, multi-tier architectures[cite: 1] using Deployments, Services, and Routes linked seamlessly together via internal DNS. | **The Sidecar Pattern:** Containers inside the *same* Pod share the network namespace and storage volumes. Use this pattern when you need a primary application container and a secondary "sidecar" container (e.g., a logging agent that reads the primary app's log files and forwards them to a central server). Frontend and database containers should never be crammed into the same Pod in a production or exam scenario. |
| **Use application health monitoring**[cite: 1] | Instructors test heavily here[cite: 1]. You must configure these flawlessly.<br>• **Liveness:** Restarts the container if deadlocked[cite: 1].<br>• **Readiness:** Stops traffic until ready[cite: 1]. | `oc set probe deploy/<name> --readiness --get-url=http://:8080/health --initial-delay-seconds=10`<br>`oc set probe deploy/<name> --liveness --tcp-socket=8080`<br>`oc set probe deploy/<name> --remove`[cite: 1] | Preventing downtime by ensuring OpenShift doesn't send user traffic to a container that hasn't finished booting up yet[cite: 1]. | **Probe Tuning Parameters:** Probe parameters are critical. `initialDelaySeconds` pauses the probe from firing immediately, giving the app time to start. `timeoutSeconds` dictates how long the probe waits for a response before failing. `failureThreshold` determines how many consecutive failures dictate a restart or traffic halt. A **Startup Probe** is used for legacy apps that take unpredictably long times to boot, disabling Liveness/Readiness until the app succeeds its Startup probe once. |
| **Understand basic Git usage**[cite: 1] | Understand how OpenShift pulls from specific branches[cite: 1]. `BuildConfig` objects must target the exact Git branch or commit where your latest application code resides. | `git clone <url>`[cite: 1]<br>`git checkout -b <branch>`[cite: 1]<br>`git commit -am "update"`[cite: 1]<br>`git push`[cite: 1] | Managing the source code that your Source-to-Image (S2I) builds rely on[cite: 1] to compile fresh, up-to-date images. | **Exam Workflow:** You will often be asked to fix broken code, commit it, and trigger a build. If your build fails, ensure your `BuildConfig` is pointing to the correct branch. OpenShift can generate a Git webhook URL; when a developer runs `git push`, the Git server sends a payload to this URL, instantly triggering a new OpenShift build without requiring manual intervention via the CLI. |
| **Work with Git in OpenShift deployments**[cite: 1] | **Core DO288 Concept:** S2I (Source-to-Image)[cite: 1]. It takes raw source code, injects it into a standardized builder image (which contains the OS and runtime environment), and compiles a runnable container image. | `oc new-app <builder-image>~<git-url>`[cite: 1]<br>`oc start-build <bc> --follow`[cite: 1]<br>`oc logs bc/<name> -f`[cite: 1] | Setting up foundational pipelines that turn raw developer code into running deployments automatically[cite: 1]. | **S2I Deep Dive:** S2I relies on hidden scripts inside the builder image, primarily `assemble` (which compiles the code, runs `npm install` or `mvn clean package`) and `run` (which executes the application). If `oc start-build` fails, you must append `--follow` or check `oc logs bc/<name>` to watch the `assemble` script's standard output to catch critical syntax or dependency errors. |
| **Configure OpenShift internal registry**[cite: 1] | Understand ImageStreams[cite: 1]. They act as an abstraction layer between deployments and actual physical image registries, providing automatic rollouts, stable version tracking, and instant rollbacks. | `oc get is`[cite: 1]<br>`oc tag <source-image> <project>/<is>:<tag>`[cite: 1]<br>`oc import-image <is>`[cite: 1] | Managing version control of images and caching external images locally[cite: 1] for faster, more reliable deployments. | **ImageChange Triggers:** When a new image is pushed to the internal registry (or an external one tracked by the ImageStream), the `ImageStreamTag` updates automatically. Deployments configured with an `ImageChange` trigger detect this mutation and instantly spin up new pods with the updated code. `oc import-image` can be used to synchronize an internal ImageStream with an external tag, ensuring your cluster always knows when the upstream image changes. |
| **Configure registry requirements**[cite: 1] | Know how to authenticate to secured external registries (e.g., private enterprise repositories) if your build requires pulling proprietary base images[cite: 1]. | `oc create secret docker-registry <name> --docker-server=<url> --docker-username=<user> ...`[cite: 1]<br>`oc secrets link builder <secret>`[cite: 1]<br>`oc secrets link default <secret> --for=pull` | Ensuring the OpenShift cluster has the necessary credentials to pull required dependencies and base images[cite: 1]. | **Service Account Mapping:** You must link the secret to the `builder` ServiceAccount if OpenShift needs to pull a base image *during* an S2I code compilation process. Conversely, link it to the `default` ServiceAccount (using the `--for=pull` flag) if you are simply deploying an existing, pre-built image to run a pod. Remember that Secrets are namespace-bound; a secret created in Project A cannot be used by a builder in Project B. |
| **Manage applications using the web console**[cite: 1] | Be able to fluently navigate the Developer Perspective to visualize Topology, inspect pod logs, and manually trigger builds[cite: 1]. | Navigate to **Topology** view, click resources to view **Metrics**, **Logs**, and **Events**[cite: 1]. | Troubleshooting complex pod crashes visually when CLI output is dense or overwhelming[cite: 1]. | **Visual Troubleshooting Advantage:** While the CLI is generally faster for resource creation, the Web Console is vastly superior for rapid diagnostics. The "Events" tab on a crashing pod will immediately reveal issues like `ErrImagePull` (due to bad registry credentials), `OOMKilled` (Out of Memory - meaning you need to adjust resource limits), or failing Liveness probes in plain text, saving you from parsing complex output under time pressure. |
