### Red Hat OpenShift Container Platform (DO288) Master Exam Guide

| Objective | Instructor Expectations & Accurate Behavior | Essential `oc` Commands & Tasks | Exam / Practical Usage | Key Notes & Advanced Exam Mechanics |
| :--- | :--- | :--- | :--- | :--- |
| **Work with Red Hat OpenShift Container Platform** | Navigate the API natively without external documentation. Understand core resource connections. | `oc login -u <user> -p <pass> <url>`<br>`oc api-resources`<br>`oc explain deployment.spec.template.spec`<br>`oc get events` | Logging into the cluster, validating context, and looking up YAML definitions on the fly. | **The Core Exam Lifeline:** You will not have internet access during the exam. `oc explain` acts as your built-in documentation for YAML syntax. `oc get events` is the optimal first step for troubleshooting, displaying the cluster's real-time operations. |
| **Create and work with multiple OpenShift projects** | Treat projects as strict boundaries. Deploying an app in the wrong project during an exam guarantees a failure for that task. | `oc new-project <name>`<br>`oc project <name>`<br>`oc status`<br>`oc delete project <name>` | Creating isolated namespaces and verifying your active project before running commands. | **Exam Trap:** You will juggle multiple projects. Always run `oc project <target-name>` as your absolute first step for any task. Projects are standard Kubernetes namespaces pre-configured with default ServiceAccounts and security policies. |
| **Create and deploy single-container applications** | Deploy from images, source code, or YAML. Understand the separation between internal routing and external ingress. | `oc new-app <image> --name=<app> -e KEY=val`<br>`oc scale deploy/<name> --replicas=3`<br>`oc expose svc/<name> --port=<port>`<br>`oc create route edge --service=<name>` | Rapidly deploying stateless servers, injecting environment variables, and safely exposing ports. | **Traffic Architecture:** External traffic hits the **Route** ➔ passes to the **Service** (Internal Load Balancer) ➔ distributed to the **Pods**. Modifying an environment variable (`oc set env`) automatically triggers a new pod rollout. |
| **Create and deploy multi-container applications** | Deploy decoupled components (e.g., frontend and database) as *separate* Pods. Only use multi-container Pods for tightly coupled processes. | `oc new-app` (run for each distinct component)<br>`oc apply -f <manifest.yaml>`<br>`oc get pods -o wide` | Deploying multi-tier architectures linked seamlessly together via internal DNS. | **The Sidecar Pattern:** Containers inside the *same* Pod share the network namespace (localhost) and storage volumes. Use this for a primary app and a secondary agent (like a logging sidecar). Never cram a frontend and database into the same Pod. |
| **Use application health monitoring** | Configure probes flawlessly.<br>• **Liveness:** Restarts container if deadlocked.<br>• **Readiness:** Stops traffic until ready. | `oc set probe deploy/<name> --readiness --get-url=http://:8080/health --initial-delay-seconds=10`<br>`oc set probe deploy/<name> --liveness --tcp-socket=8080`<br>`oc set probe deploy/<name> --remove` | Preventing downtime by ensuring OpenShift doesn't route traffic to a booting container. | **Probe Tuning:** `initialDelaySeconds` prevents the probe from firing immediately while the app boots. `timeoutSeconds` dictates how long it waits for a response. **Startup Probes** are used for slow legacy apps, disabling Liveness/Readiness until the app boots successfully once. |
| **Understand basic Git usage** | Understand how `BuildConfig` objects target exact Git branches or commits to pull application code. | `git clone <url>`<br>`git checkout -b <branch>`<br>`git commit -am "update"`<br>`git push` | Managing source code that Source-to-Image (S2I) pipelines rely on to compile fresh images. | **Exam Workflow:** You will often need to fix broken code, commit it, and trigger a build. If a build fails, ensure your `BuildConfig` points to the correct branch. Git webhooks can automate this, triggering OpenShift builds instantly on a `git push`. |
| **Work with Git in OpenShift deployments** | **Core DO288 Concept:** S2I (Source-to-Image). It injects raw code into a builder image (OS + runtime environment) to compile a runnable container. | `oc new-app <builder-image>~<git-url>`<br>`oc start-build <bc> --follow`<br>`oc logs bc/<name> -f` | Setting up automated pipelines that turn raw developer code into running deployments. | **S2I Deep Dive:** S2I relies on hidden scripts inside the builder image: `assemble` (compiles code, runs `npm install`, etc.) and `run` (executes the app). If `oc start-build` fails, use `--follow` to watch the `assemble` script's output for syntax or dependency errors. |
| **Configure OpenShift internal registry** | Understand ImageStreams. They act as an abstraction layer between deployments and physical image registries. | `oc get is`<br>`oc tag <source-image> <project>/<is>:<tag>`<br>`oc import-image <is>` | Caching external images locally and managing version control for reliable deployments. | **ImageChange Triggers:** When a new image is pushed to the registry, the `ImageStreamTag` updates automatically. Deployments configured with an `ImageChange` trigger detect this and instantly spin up new pods. `oc import-image` syncs internal ImageStreams with external tags. |
| **Configure registry requirements** | Authenticate to secured external registries (e.g., private enterprise repositories) to pull proprietary base images. | `oc create secret docker-registry <name> --docker-server=<url> --docker-username=<user> ...`<br>`oc secrets link builder <secret>`<br>`oc secrets link default <secret> --for=pull` | Ensuring the OpenShift cluster has credentials to pull required dependencies. | **Service Account Mapping:** Link the secret to the `builder` ServiceAccount if OpenShift pulls a base image *during* an S2I compilation. Link it to the `default` ServiceAccount (with `--for=pull`) to simply deploy a pre-built image. Secrets are strict to their specific project namespace. |
| **Manage applications using the web console** | Fluently navigate the Developer Perspective to visualize Topology, inspect logs, and manually trigger builds. | Navigate to **Topology** view, click resources to view **Metrics**, **Logs**, and **Events**. | Troubleshooting complex pod crashes visually when CLI output is overwhelming. | **Visual Diagnostics:** While the CLI is faster for creation, the Web Console is superior for rapid troubleshooting. The "Events" tab on a crashing pod will immediately reveal issues like `ErrImagePull` (bad registry credentials) or `OOMKilled` (Out of Memory) in plain text. |
